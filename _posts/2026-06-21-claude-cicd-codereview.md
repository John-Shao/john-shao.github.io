Claude Code Review 使用多个 AI 代理并行分析代码更改，在发现问题的代码行上发布内联评论。通过完整代码库的上下文，它可以捕获逻辑错误、安全漏洞、破损的边界情况和微妙的回归问题。

> 📌 **可用性**：Code Review 处于研究预览阶段，仅适用于 Team 和 Enterprise 订阅。对于启用了 Zero Data Retention 的组织，此功能不可用。

## 审查工作原理

一旦管理员为您的组织启用 Code Review，审查将在 PR 打开时、每次推送时或手动请求时触发，具体取决于存储库的配置行为。在任何模式下，注释 `@claude review` 可以在 PR 上启动审查。

### 审查过程

当审查运行时，多个代理在 Anthropic 基础设施上并行分析差异和周围代码。每个代理寻找不同类别的问题，然后通过验证步骤检查候选项是否与实际代码行为相符，以过滤掉误报。结果被去重、按严重程度排序，并作为内联评论发布在发现问题的特定行上。

- **平均完成时间**：20 分钟以内
- **成本**：约 $15-25/次，根据 PR 大小和复杂性浮动
- **管理员监控**：通过分析仪表板查看审查活动和支出

### 严重程度级别

每个发现都标有严重程度级别：

| 标记 | 严重程度 | 含义                                 |
| :--- | :------- | :----------------------------------- |
| 🔴    | 重要     | 应在合并前修复的错误                 |
| 🟡    | 小问题   | 轻微问题，值得修复但不阻止           |
| 🟣    | 预先存在 | 代码库中存在但不是由此 PR 引入的错误 |

发现包括可折叠的扩展推理部分，您可以展开以了解 Claude 为什么标记该问题以及它如何验证问题。

### 对发现进行评分和回复

Claude 的每条审查评论都已附加 👍 和 👎，因此两个按钮都会在 GitHub UI 中出现，以便一键评分：

- **👍 有用**：帮助改进代码
- **👎 无用/错误**：误报或不相关

Anthropic 在 PR 合并后收集反应计数，并使用它们来调整审查者。

**处理发现的流程**：
1. 修复代码并推送
2. 如果 PR 订阅了推送触发的审查，下一次运行会自动解决线程
3. 要获得立即重新审查，注释 `@claude review once`

### 检查运行输出

每次审查都会填充 **Claude Code Review** 检查运行，与您的 CI 检查一起出现。展开其 **Details** 链接以查看：

- 按严重程度排序的每个发现摘要
- 文件和行号
- 发现类别和摘要

**示例**：

| 严重程度 | 文件:行                   | 问题                                       |
| -------- | ------------------------- | ------------------------------------------ |
| 🔴 重要   | `src/auth/session.ts:142` | 令牌刷新与登出竞争，导致过期会话保持活跃   |
| 🟡 小问题 | `src/auth/session.ts:88`  | `parseExpiry` 在格式错误的输入上静默返回 0 |

发现会以多种方式呈现：
- **Files changed 选项卡**中的内联注释
- **检查运行 Details** 中的汇总表
- 审查正文的摘要

> 💡 **中立结论**：检查运行始终以中立结论完成，因此永远不会通过分支保护规则阻止合并。

### Code Review 检查的内容

默认情况下，Code Review 专注于正确性：会破坏生产的错误，而不是格式偏好或缺失的测试覆盖。您可以通过向存储库添加指导文件来扩展其检查范围。

## 设置 Code Review

管理员为组织启用一次 Code Review，并选择要包含的存储库。

### 设置步骤

**1. 打开 Claude Code 管理员设置**

转到 [claude.ai/admin-settings/claude-code](https://claude.ai/admin-settings/claude-code) 找到 Code Review 部分。您需要对 Claude 组织具有管理员访问权限，并有权在 GitHub 组织中安装 GitHub Apps。

**2. 开始设置**

点击**设置**。这将开始 GitHub App 安装流程。

**3. 安装 Claude GitHub App**

按照提示将 Claude GitHub App 安装到您的 GitHub 组织。该应用请求这些存储库权限：

- **Contents**：读写
- **Issues**：读写  
- **Pull requests**：读写

Code Review 使用对内容的读取访问权限和对 pull request 的写入访问权限。

**4. 选择存储库**

选择要为 Code Review 启用的存储库。如果您看不到存储库，请确保在安装期间为 Claude GitHub App 提供了对其的访问权限。您可以稍后添加更多存储库。

**5. 设置审查触发器**

设置完成后，Code Review 部分在表格中显示您的存储库。对于每个存储库，使用**审查行为**下拉菜单选择何时运行审查：

| 触发器            | 说明                                                   | 成本                   |
| ----------------- | ------------------------------------------------------ | ---------------------- |
| **PR 创建后一次** | 当 PR 打开或标记为准备审查时运行一次审查               | 最低                   |
| **每次推送后**    | 在每次推送到 PR 分支时运行审查，在 PR 演变时捕获新问题 | 最高（按推送次数倍增） |
| **手动**          | 仅当有人在 PR 上注释 `@claude review` 时启动           | 可控                   |

存储库表还显示每个存储库基于最近活动的平均审查成本。

### 验证设置

打开测试 PR。如果您选择了自动触发器，在几分钟内会出现名为 **Claude Code Review** 的检查运行。如果您选择了手动，在 PR 上注释 `@claude review` 以启动第一次审查。

## 手动触发审查

两个注释命令按需启动审查。无论存储库的配置触发器如何，两者都有效。

### 触发命令

| 命令                  | 作用                                     |
| :-------------------- | :--------------------------------------- |
| `@claude review`      | 启动审查并将 PR 订阅到今后的推送触发审查 |
| `@claude review once` | 启动单次审查，不订阅未来推送             |

### 使用指南

当您想要对 PR 的当前状态获得反馈但不希望每次后续推送都产生审查时，使用 `@claude review once`。这对于具有频繁推送的长期运行 PR 很有用。

**重要**：
- 将注释放在 PR 的**顶级评论**中，而不是差异行上
- 您必须对存储库具有所有者、成员或协作者访问权限
- PR 必须打开
- 手动触发在**草稿 PR** 上也能运行

### 处理排队的审查

如果该 PR 上已有审查正在运行，请求将排队等待进行中的审查完成。您可以通过 PR 上的检查运行监控进度。

## 自定义审查

Code Review 从您的存储库读取两个文件来指导它标记的内容。它们在如何强烈影响审查方面有所不同：

### CLAUDE.md

**用途**：共享项目说明，Claude Code 用于所有任务，不仅仅是审查。

**作用**：
- Code Review 将其作为项目上下文读取
- 新引入的违规标记为小问题
- 支持目录层次结构（子目录的规则仅适用于该路径下的文件）

### REVIEW.md

**用途**：仅审查说明，直接注入到审查管道中每个代理的系统提示中。

**优先级**：作为最高优先级指令块，优先于默认审查指导

**重要**：`REVIEW.md` 是纯说明，不支持 `@` 导入语法展开。将所有规则直接放在文件中。

### 可调整的内容

#### 严重程度

为您的存储库重新定义 🔴 重要的含义。默认校准针对生产代码；文档存储库、配置存储库或原型可能想要更窄的定义。

示例：
```markdown
# 审查说明

## 重要在这里的含义

保留重要用于会破坏行为、泄露数据或阻止回滚的发现：
- 不正确的逻辑
- 无范围的数据库查询
- 日志或错误消息中的 PII
- 不向后兼容的迁移

风格、命名和重构建议最多是小问题。
```

#### 小问题数量

限制单次审查发布的 🟡 小问题评论数量。这使审查可操作。

示例：
```markdown
## 限制小问题

每次审查最多报告五个小问题。如果您发现了更多，
请在摘要中说"加上 N 个类似项目"而不是内联发布它们。
```

#### 跳过规则

列出 Claude 应该不发布任何发现的路径、分支模式和发现类别。

常见候选：
- 生成的代码
- lockfiles
- 供应商依赖
- 机器创作的分支
- CI 已强制执行的内容（linting、拼写检查等）

示例：
```markdown
## 不要报告

- CI 已经强制执行的任何内容：lint、格式化、类型错误
- `src/gen/` 下生成的文件和任何 `*.lock` 文件
- 故意违反生产规则的仅测试代码
```

#### 存储库特定检查

添加您想在每个 PR 上标记的规则。因为 `REVIEW.md` 被注入为最高优先级，这些比长 `CLAUDE.md` 中的相同规则更可靠地着陆。

示例：
```markdown
## 始终检查

- 新 API 路由有集成测试
- 日志行不包括电子邮件地址、用户 ID 或请求正文
- 数据库查询的范围限定为调用者的租户
```

#### 验证标准

在发布发现类别之前需要证据。

示例：
```markdown
行为声明需要源中的 `file:line` 引用，而不是从命名推断。
```

### 完整示例 REVIEW.md

```markdown
# Code Review 指导

## 严重程度定义

🔴 **重要**：应在合并前修复
- 生产错误、数据泄露、回滚阻止
- 不正确的逻辑、无范围查询、PII 暴露
- 破坏性迁移

🟡 **小问题**：值得修复
- 风格、命名、重构建议
- 文档改进

## 限制小问题数量

每次审查最多 5 个小问题。额外的问题在摘要中统计。

## 跳过检查

- `src/gen/` 中的生成代码
- `*.lock` 文件
- 测试中故意违反的规则
- CI 已强制执行的（linting、格式化）

## 必检项

- ✅ 新 API 路由有集成测试
- ✅ 日志行无 PII（邮箱、用户 ID）
- ✅ 数据库查询已范围限制
```

> 💡 **最佳实践**：保持 `REVIEW.md` 简洁专注。将常规项目上下文留在 `CLAUDE.md` 中。

## 定价

Code Review 根据令牌使用情况计费。每次审查的成本示意：

- **平均成本**：$15-25/次
- **变量因素**：PR 大小、代码库复杂性、验证问题数量

### 成本影响因素

您选择的审查触发器直接影响总成本：

| 触发器            | 成本影响                               |
| ----------------- | -------------------------------------- |
| **PR 创建后一次** | 每个 PR 运行一次                       |
| **每次推送后**    | 在每次推送时运行，成本随推送次数倍增 ⚠️ |
| **手动**          | 仅在明确请求时产生成本 ✅               |

### 计费说明

- **使用额度**：计入单独的使用额度，不计入您的计划包含的使用
- **组织配置**：无论您的组织是否为其他 Claude Code 功能使用 Amazon Bedrock 或 Google Vertex AI，成本都会出现在您的 Anthropic 账单上
- **支出上限**：转到 [claude.ai/admin-settings/usage](https://claude.ai/admin-settings/usage) 为 Claude Code Review 服务配置每月支出上限

### 监控成本

三种方式监控支出：

1. **分析仪表板**：[claude.ai/analytics/code-review](https://claude.ai/analytics/code-review)
   - 每日审查 PR 计数
   - 每周支出
   - 因开发人员解决问题而自动解决的审查评论计数
   - 存储库分解

2. **管理员设置**：存储库表显示每个存储库的平均审查成本

3. **账单**：仪表板成本数字是估计值；准确的支出见 Anthropic 账单

## 故障排除

审查运行是尽力而为的。失败的运行永远不会阻止您的 PR，但它也不会自动重试。

### 重新触发失败或超时的审查

当审查基础设施遇到内部错误或超过时间限制时，检查运行会显示：
- **Code review encountered an error** 或 
- **Code review timed out**

结论仍然是中立的，因此没有任何东西阻止您的合并。

**恢复方法**：
1. 在 PR 上注释 `@claude review once` 启动新审查
2. 或推送新提交（如果 PR 订阅了推送触发审查）

> ⚠️ 注意：GitHub 检查选项卡中的**重新运行**按钮不会重新触发 Code Review。改用注释命令或新推送。

### 审查未运行，显示支出上限消息

当您的组织的每月支出上限达到时，Code Review 在 PR 上发布单条评论，解释审查被跳过。

**恢复步骤**：
- 审查在下一个计费周期开始时自动恢复
- 或当管理员在 [claude.ai/admin-settings/usage](https://claude.ai/admin-settings/usage) 提高上限时立即恢复

### 查找未显示的问题

如果检查运行标题说发现了问题但您在差异上看不到内联审查评论，请在这些位置查看：

1. **检查运行 Details**
   - 在检查选项卡中的 Claude Code Review 检查旁边点击 **Details**
   - 严重程度表列出每个发现及其文件、行和摘要

2. **Files changed 注释**
   - 打开 **Files changed** 选项卡
   - 发现呈现为直接附加到差异行的注释

3. **审查正文**
   - 如果推送到 PR 时发现引用不再存在的行
   - 这些出现在 **Additional findings** 标题下

## 在本地审查差异

`/code-review` 命令在您的终端中审查差异，无需安装 GitHub App。

### 基本用法

在任何 Claude Code 会话中运行：

```bash
/code-review
```

**功能**：
- 报告正确性错误
- 进行代码简化和效率审查
- 默认涵盖您分支相对于其上游的所有提交加上工作树中的未提交更改

### 常用选项

| 选项                | 说明                                     |
| ------------------- | ---------------------------------------- |
| `--comment`         | 将发现作为内联 PR 评论发布               |
| `--fix`             | 在审查后将发现应用到您的工作树           |
| `--effort high/max` | 提供更广泛的覆盖范围（包括不确定的发现） |

### 指定审查范围

默认审查不同的目标：

- **文件路径**：`/code-review src/app.ts`
- **PR 编号**：`/code-review #42`
- **分支名称**：`/code-review my-feature`
- **引用范围**：`/code-review main...my-feature`

### Ultra Review

运行更深入的云端审查：

```bash
/code-review ultra --fix
```

**范围**：您当前分支与存储库默认分支的差异，加上工作树中的任何未提交和暂存更改。

> 💡 **版本说明**：
> - v2.1.147 之前：命名为 `/simplify`
> - v2.1.154+：`/simplify` 运行仅清理审查，应用修复而不寻找错误
